I authored a stdlib-`unittest` held-out suite for `preview-pipeline` in `suite/`, testing `run_preview(text, fmt, ir) -> {"preview_rows", "errors"}` from the **spec alone** (no implementation exists yet).

**Files written** (`suite/`):
- `_preview_support.py` — lazily-binding loaders (no `eaitl` imports at module load, so it imports cleanly on base) + the core `assert_preview_composes` differential oracle + `find_json_source` (source-encoding hedge).

**Design choice that de-risks the unknowns:** the required `source-introspection` dependency isn't in the base checkout, and its exact internals (sample key name, JSON source encoding, export location) aren't fully pinned by the spec. So the backbone is **differential composition** — whatever `introspect_source` does, `run_preview` must do the same — which is robust to all of that, plus resilient loaders and a multi-encoding source finder so I never false-fail correct-but-different implementations.
