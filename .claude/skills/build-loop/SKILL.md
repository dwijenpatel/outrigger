# Build loop — one firing

You are the orchestrator for one operator-started firing. Deterministic machinery does everything it can; you supply judgment only where scripts cannot. **Mandatory invocations below are phase-gated: run them at the stated point, every time — never rely on remembering.**

## 0. Start (mandatory, in order)

0. Permission mode (I30, P3v2-14): a firing is arbitrary-Bash heavy — in `default`/`manual` mode it floods the operator with permission prompts almost immediately. After your first message (the statusline dump self-refreshes per message), check `harness.loop.permission_mode('state/statusline-dump.json')`: a mode outside `harness.loop.FIRING_PERMISSION_MODES` (`auto`, `bypassPermissions`) means **STOP before any real work** — tell the operator to flip with Shift+Tab or relaunch `claude --permission-mode auto`, then re-invoke this skill. `None` = the field is absent on this build (verified missing on 2.1.201) or no dump yet: advisory only — say so and proceed; the kickoff's launch flag is the prevention. Note the mode cannot be changed programmatically — it is operator-only.
0b. Plan readiness: `python3 -m harness.planning ready --plan-dir plan --snapshot state/plan-snapshot.json --vault-config harness/config/vault-isolation.json`. Exit != 0 means **no firing** — no ratified plan (run `plan-build`) or no configured vault (`python3 -m harness.vault configure --vault-path /abs/path-outside-repo`). Never author or patch a plan from inside a firing.
1. Acquire the run marker: `harness.loop.acquire_run_marker(state/run.marker, <firing-id>)`. A live-owner refusal means STOP — concurrent firings are operator-arbitrated. Immediately write `state/closure-hook.json` via `harness.loop.closure_hook_config(...)` — the registered Stop hook reads it, and a live firing without it blocks every stop (fail-closed).
2. Verify gates: `python3 -m harness.selftest` must pass, then the composition proof: `python3 -m harness.smoketest` must exit 0 (I5 — one zero-quota walk of this whole step sequence in a scratch clone, ~2s; a machinery sync that broke the *sequence* while every module stayed green is exactly what it catches). Either failing = fail closed: stop, report.
2b. Vault config: `python3 -m harness.vault check` must exit 0. Exit 2 = unconfigured or drifted — STOP; the operator sets it with `python3 -m harness.vault configure --vault-path /abs/path-outside-repo` (never hand-edit the config). Use the checked config's `vault_path` for every `worker_settings`/gate call below.
3. Check skill inventory: `python3 -c "from harness import loop; r = loop.skill_budget_check(['.claude/skills']); print(r['why']); raise SystemExit(0 if r['ok'] else 1)"` — over budget means skills are silently vanishing; stop and report.
3b. Preflight quota sources: `python3 -m harness.governor --preflight --statusline-json state/statusline-dump.json [--oauth-json ...]` (missing source files are skipped rungs, not errors). **No dump yet in an interactive session? It is produced by the statusline AFTER each of your messages — finish this message, then re-run the preflight once at the next step before walking the bootstrap ladder** (first-tick race; the dump self-refreshes for the rest of the firing). Exit 3 after the re-check = **conservative mode**: tightened thresholds, cheap-serial only, no concurrency admissions — or stop for operator ack. Estimate-rung data alone never clears a preflight. **No live rung? Match the fix to the operator's environment (P2-8):** the statusline shim fires only in **terminal CLI** sessions — desktop-app/web workflows have no statusline surface, ever; do not wait or watch for a dump there. Preference order: (a) terminal workflows — the registered statusline shim self-feeds the dump; (b) desktop workflows — the operator runs `tools/oauth_usage_refresh.sh <this-repo>/state/oauth-usage.json` in any terminal (continuously fresh OAuth rung, token never enters this session); (c) one-shot `CLAUDE_OAUTH_TOKEN=... tools/oauth_usage_fetch.sh > state/oauth-usage.json` (stales in 600s); (d) explicit operator-acked assumption for conservative-serial admissions only: `--assume-occupancy 0.3 --acked-by <operator>` — attributed and sticky-`~`, never a silent default.
4. **Resume from artifacts only:** `harness.loop.resume_context(...)`. Act on the bundle (git facts, reconciled digest, pending events). Never resume from a prior session's summary — logs and notes are claims, not evidence. Drain `pending_events` before new work.

## Per tick (mandatory)

0. Pause request: `harness.loop.pause_requested(state/pause.request)` non-None → **acknowledge, drain, pause** (I23): first `harness.loop.acknowledge_pause('state/pause.request', 'state/pause.ack', draining=[<in-flight task:role ids>])` — the ack is the operator's receipt, written BEFORE the drain starts; then admit nothing new, let in-flight attempts run to their handoff (killing an attempt = redoing it), then the `build-pause` orchestrator sequence (park + write-ahead records, resume marker, release run marker, clear the request *after* pausing). **Check the flag at every stage boundary too** — after each worker handoff, panel return, and gate exit, not only at tick start. The flag file is the operator's only responsive channel into a live firing: a message typed into the firing session starves in the queue until the turn yields (P3v2-8).
1. Governor between tasks: `python3 -m harness.governor ...` (statusline dump file → oauth doc → run-log estimate). `pause` → clean pause (same sequence). `degrade` → profile-minimum panels, no new heavy admissions.
2. Scheduler: `harness.scheduler.tick(...)` with current occupancy, estimates, slots — and **reset headroom (I17)**: compute `governor.fraction_rate(decisions, binding_window, since_ts=<this firing's run-marker acquired_at, as epoch>)` over `state/governor-log.jsonl` (I25: burn rate is per-firing — readings that predate this firing's marker never count, so a fresh firing stays conservative until ~10 min of its own readings) plus the decision's `resets_in_s`, feed both to `governor.reset_headroom(...)`, and pass `reset_headroom_clears=` its verdict. A binding window about to reset with projected-at-reset occupancy under pause waives tail-capping (degrade/pause still bind); anything unknown stays conservative. If the live feed lacks the weekly `resets_at`, the operator sets `weekly_reset_anchor` in `harness/config/window-resets.json` (any past weekly reset instant off `/usage`; it rolls +7d) — 5-hour resets come from live rungs only. Start what the tick says; record `held_unsafe`/`deferred` reasons in the ledger digest — never override its admission decisions to "just fit one more in". After an admitting tick, write the admission stamp: `harness.interlocks.write_admission_stamp(state/admission-stamp.json, {task_id, tick})` — the spawn interlock refuses worker spawns without a fresh one.
3. Per admitted task: spawn params come from `harness.spawncheck.profile_spawn_params(profile, regime=task.get('regime'))` (I18: the regime conditions the implementer tier; efforts are uniform xhigh, `max` reserved for escalation) — **test-author and validators use the base `(model, effort)`; the implementer uses the `implementer` block** (it may start a tier below: its failures are loud and self-correcting via the ladder; validation-side weakness is silent and never downgrades by config). **Workers are headless one-shot processes (I26)** — the one spawn path where every knob verifiably binds (per-spawn `--model` concrete id + `--effort` measured-applied; per-worker settings). Per worker: give it a home (`git worktree add` for the implementer; a scratch checkout for test-author/validators), bind its policy with `harness.loop.write_worker_overlay(worktree, vault_path)` (§7 layer-6 divergence — gitignored, so `require_clean` stays green), build the argv with `harness.loop.headless_worker_cmd(prompt, model, effort=..., system_prompt=<role body from .claude/agents/<role>.md>, json_schema_path=harness/config/schemas/<handoff|verdict>.json, disallowed_tools=["Edit","Write","NotebookEdit"] for validators)`, and run it under a **wall-clock deadline** via `harness.loop.run_headless_worker(argv, cwd=<worktree>, env=harness.loop.headless_env(...), timeout_s=<3× the profile's P95 wall estimate; default 2700>)` inside a **background** Bash call (I29/P3v2-13: `--max-turns` never fires on a pre-compute hang — a quota-stalled worker sat 38 min at 0% CPU until manually killed; the deadline kills the process group and returns the death structured) — the admission-stamp spawn interlock governs the command; a foreground worker deafens the orchestrator (I23/P3v2-8); between polls, re-check `state/pause.request`. Harvest with `harness.loop.parse_worker_result` (fence-tolerant, P3v2-10; `parsed=None` = contract violation → ladder, not a crash); classify deaths and `timed_out` via `failures.classify(stderr, extra_patterns=failures.load_patterns(loop.HEADLESS_FAILURE_PATTERNS))`. *(Environment fallback: if headless spawns are unavailable, Agent-tool spawns are legal — model-only, effort advisory, overlay non-binding — record the downgrade in the firing digest.)* Sequence per task: spawn **test-author** (spec-only; move outputs to the vault, re-record the manifest); on its return run `harness.schemas.ambiguity_blockers(handoff, task_id, profile)` — entries marked `corpus_covers: "both"` are already discharged and never block (I20); non-empty on high/critical means **park the task on those blockers now**, before the implementer spawns, via the **operator-adjudication protocol** below. Then the **implementer** (scoped spec + `harness.loop.LessonsCorpus.select_for_task` injection), then the blind **validator panel** (fresh contexts, spec+diff only, one lens each).
4. Feed each worker step to `harness.liveness.Vitals`; on a park recommendation, park via the operator-adjudication protocol and continue other work.
5. Merge only through the gate: `harness.gate.run_gate(...)` with verdict dir, floors config, vault path, evidence dir, `stamp_dir=state/gate-stamps` (a PASS writes the stamp the merge interlock demands; without it `git merge` is hook-blocked during the firing), `required_steps_path=harness/config/gate-required-steps.json` (the manifest turns omitted inputs into fail-closed refusals for the task's profile), and **`heldout_cmd` (I14): read it from `plan/conventions.json` when present (P3v2-4 — the plan pins the project's held-out run convention; e.g. `python3 -m pytest .heldout -q`), else from the kickoff/resume marker; the gate materializes the vault corpus into the clean checkout and runs it itself — never hand-copy held-out tests or run them outside the gate (leakage metering + H7 evidence routing live there)**. Risk floors run for every profile — a task whose diff touches floored paths above its profile bounces as mis-tagged; that is a plan problem (re-profile + re-ratify), never a reason to drop the floors input. The gate's exit status decides — **check the real exit status, never a piped tail of it**.
6. Handle worker infrastructure errors via `harness.failures.classify` → continue / backoff / abort per the taxonomy. Agent-reported FAILs are escalation signals at a fresh worker boundary, not errors. **Escalation ladder (I26 — real on the headless path):** attempt 2 = **fresh worker, same tier, `effort="max"`** (measured-applied: benchmark 2026-07 low→max scales thinking 2.4–4.2×) **+ sharpened spec-grounded feedback** (the gate's scrubbed failure evidence — the feedback half alone landed GL1 attempt 2); attempt 3+ = **tier up** (implementer only; validation-side tiers never move by config), effort back to the profile base. **Tier-up saturates at `capable`/opus while fable is machinery-reserved (I28)** — an attempt-3 tier-up from capable changes nothing: park for the operator instead. A 429/credit-exhaustion or backoff-stall abort under an over-degrade window is **window pressure, not a model problem** — model substitution does not dodge it (P3v2-12/13): park until the window clears. On the Agent-tool *fallback* path there is no effort knob (P3v2-6/I24): the recorded `effort` is telemetry, and the ladder is feedback → tier only — never plan an escalation on a rung the spawn path can't actuate (design §5.3 + probe addendum 2026-07-05).
7. Append every outcome to the run-log and ledger event log **before** advancing any cursor/marker (write-ahead). Each run-log record carries the **spawn params you requested** — `role`, `tier`, **`model` (the concrete id from `spawncheck`'s resolved params)**, `effort`, `profile` — never a worker's self-report. A worker that dies or parks mid-task still gets its record (`outcome: "aborted"`/`"parked"`) so the routing choice is never lost.

## Operator adjudication (mandatory whenever a human decision is needed)

Every operator decision — ambiguity blockers, gate `needs_adjudication`, park recommendations, any E3 card — follows this order, no exceptions (P3v2-1: a question that lived only in a session's context stalled a firing — first measured 8h25m, clock-corrected to ≲85 min by P3v2-3, unbounded either way — while the ledger claimed `in_progress`):

1. **Card first (write-ahead).** Write the blocker card (schema `blocker.json`) to `state/blockers/<task>-<kind>.json` with `kind`, `asked_at` (ISO instant), repro, ≥2 options, recommendation — **before** any interactive question. The card IS the question; an interactive prompt is only one delivery vehicle for it. A session that dies while waiting loses nothing.
2. **Ledger truth.** Append the `parked` status (note names the card path) **before** asking. An `in_progress` task with no live worker is a claims-vs-evidence violation — and `parked→in_progress` timestamps are the operator-wait telemetry (O2).
3. **Notify.** `tools/notify_operator.sh "<task>: decision needed (state/blockers/<file>)"` — best-effort, never a gate.
4. **Only then ask — and only if idle.** If other admissible work exists, do NOT ask interactively: continue the loop and let the operator answer cards on their own clock. Ask interactively (one question, the card's options verbatim) only when the parks leave **nothing** admissible (e.g. the serial root parked). The two answer paths are equivalent: an interactive answer and a hand-edited card resolve identically.
5. **Record the resolution.** Write `resolved: {decision, by, at}` (ISO instant, not a bare date) into the card, append the ledger resume (`parked→in_progress`, note names the card), then proceed. Never re-adjudicate a resolved card.

## Pipelining (while validation runs)

Author task N+1's spec + contract tests while task N validates; pre-decompose the next phase into a provisional, planning-only ledger. Look-ahead is planning-only — never speculative implementation.

## Situational (invoke when the trigger fires, not by default)

- **Output diverges and you can't see why** → stop guess-and-patch; reproduce in isolation with a fresh worker.
- **Any surprise mismatch between a summary and an artifact** → trust the artifact; re-read state via `resume_context`.
- **key_learnings arrive** → `LessonsCorpus.add` (surprising-only contract; exact-dupe is dropped).
- **A machinery change seems needed** → never edit machinery; queue a decision card (E3) and continue.

## Close (mandatory)

Phase close: batch ratification cards; regenerate evidence roll-up. Firing end: reconcile worktrees, release the run marker, leave the ledger digest as the handoff — the next firing resumes from disk, not from your summary.
