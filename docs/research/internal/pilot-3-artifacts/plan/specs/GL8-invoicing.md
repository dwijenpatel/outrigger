# GL8-invoicing — batch invoice engine + invoice lifecycle

## Context

greenlane (multi-tenant landscaping SaaS; FastAPI + Jinja2 + SQLAlchemy).
Exists: auth/tenancy (GL2), OutboxEmail table (GL3), customers/properties
(GL4), schedules with `billing_mode` (`per_visit` | `fixed_monthly`) and
`price_cents` (GL5), visits with `status` in {scheduled, completed,
skipped}, nullable `schedule_id`, `price_cents` carried on per-visit and
standalone visits (GL5/GL6). This task turns completed work into money:
**idempotent month-end batch invoicing** across both billing modes, plus
the invoice lifecycle. Money = integer cents, rendered as dollars. No tax,
no proration (plan decisions D11/D22). Month boundaries use the company's
IANA timezone only to define "today"; a month M is the calendar dates
`M-01 .. last day of M`. All screens owner-only.

## Models (add to `app/models.py`)

- `Invoice`: `id`, `company_id` (FK, indexed), `customer_id` (FK),
  `status` (`draft` | `sent` | `paid` | `void`), `created_at`, `sent_at`,
  `voided_at` (nullables). Total is always computed = sum of line
  `amount_cents` (never stored).
- `InvoiceLine`: `id`, `invoice_id` (FK, indexed), `company_id`,
  `description` (non-empty), `amount_cents` (int), `visit_id` (nullable FK,
  **unique when set** — a visit is on at most one line),
  `monthly_schedule_id` + `monthly_period` (`YYYY-MM` str; both nullable,
  set together; unique together when set — a schedule-month is billed at
  most once), `position`. A line is exactly one of: visit-backed,
  monthly-charge, or manual (all three key fields null).
  "Billed" for dedup purposes means the line's invoice status is NOT
  `void` (and, for deleted drafts, the line no longer exists).

## The engine (`app/services/billing.py`)

`generate_month(db, company_id, period: str, today: date) -> list[Invoice]`
— "generate invoices for month M":

1. **Per-visit + standalone sweep**: every visit with status `completed`,
   a non-null `price_cents`, `date ≤ last day of M`, and no live
   invoice line → line `"{service} — {date}"`, amount = its price. (The
   `≤` catches stragglers from earlier months that were never billed;
   skipped and scheduled visits never bill; fixed-monthly visits have null
   price and never bill per-visit.)
2. **Fixed-monthly sweep**: every `fixed_monthly` schedule active ≥ 1 day
   of M (`start_date ≤ last-day-of-M` AND (`end_date` null OR `end_date ≥
   M-01`)) with no live (schedule, M) line → line
   `"{service} — {MonthName YYYY}"`, amount = schedule price. **No
   proration.** Only the requested M is swept (earlier unbilled months are
   generated by running their own period).
3. Lines group into **one new draft invoice per customer** (via the visit's
   property → customer / the schedule's property → customer). Customers
   with no lines get no invoice. Ordering: monthly charges first, then
   visits by date.
4. **Idempotent**: an immediate second run for M creates zero invoices and
   zero lines. Deleting a draft (below) returns its items to the pool, so
   a later run picks them up again — that is correct, not a violation.

`generate_for_customer(db, company_id, customer_id, today) -> Invoice|None`
— ad-hoc mid-month sweep: rule 1 only (visits dated ≤ **today**), single
customer, NO monthly charges (plan decision D23). Nothing to bill → None.

## Routes (in `app/routes/invoices.py` — floored high; the `POST /customers/{id}/invoice` handler included; owner-only, mutating CSRF-checked, cross-tenant → 404)

- `POST /invoices/generate` — form `period` (`YYYY-MM`, else form error);
  runs `generate_month` with company-tz today; 303 → `/invoices` with a
  flash-style count ("3 invoices created").
- `POST /customers/{id}/invoice` — runs `generate_for_customer`; 303 → the
  new invoice, or back to the customer page with "nothing to invoice".
- `GET /invoices` — list (customer, status, total, created), status filter
  tabs; `GET /invoices/{id}` — lines, total, actions legal for its status.
- Draft editing (non-draft → 409):
  `POST /invoices/{id}/lines` — manual line (description, amount dollars —
  may be negative for a goodwill discount, total may not go below 0);
  `POST /invoices/{id}/lines/{line_id}/edit` — description/amount of ANY
  line type; edits to a visit-backed line keep the `visit_id` link;
  `POST /invoices/{id}/lines/{line_id}/delete` — removes the line (a freed
  visit/schedule-month returns to the unbilled pool);
  `POST /invoices/{id}/delete` — delete the whole draft and its lines.
- `POST /invoices/{id}/send` — draft → sent (≥ 1 line required, else 409):
  stamps `sent_at`, freezes it (all line mutations now 409), writes an
  OutboxEmail (`kind="invoice"`, to the customer's email if set else
  skipped silently, body = plain-text line summary + total).
- `POST /invoices/{id}/void` — draft or sent → void; paid → 409. Voiding
  releases every visit and schedule-month on it for future billing.

**Standing invariant:** the project is typed strict on both sides — mypy config is pinned in `pilot/greenlane/pyproject.toml` and strict JSDoc/tsc config in `pilot/greenlane/tsconfig.json`. Fully annotate all new application code: `python -m mypy app` and `npm run typecheck` (both from `pilot/greenlane/`) must stay at zero errors; any `.js` you add under `app/static/` must be JSDoc-typed.

## Acceptance criteria

1. Fixture month: per-visit schedule with 3 completed + 1 skipped +
   1 scheduled visit, a fixed-monthly schedule (visits completed, null
   price), a standalone completed visit, all one customer → ONE draft
   invoice with 3 + 1 + 1 = 5 lines and the exactly right total; skipped/
   scheduled visits absent; the fixed-monthly schedule contributes its
   monthly price once, not per visit.
2. Second `generate_month` run for the same period: zero new invoices/
   lines. Next month's run bills the monthly schedule again (new period)
   but not the already-billed visits.
3. A completed visit from LAST month never billed → included by this
   month's run (date ≤ end of M). A visit completed but dated next month →
   excluded.
4. Mid-month schedule start (start_date = M-20) → full monthly price for M
   (no proration). Schedule ended before M-01 → no charge for M.
5. Delete-draft → re-run recreates equivalent lines; void-sent → re-run
   recreates them; **paid invoices' items never return to the pool**.
6. Send freezes: line add/edit/delete on a sent invoice → 409; send writes
   exactly one outbox row per send; lineless send → 409.
7. `generate_for_customer` picks up only that customer's completed
   visits dated ≤ today and never creates monthly-charge lines.
8. Crew → 403 everywhere; cross-tenant invoice/customer ids → 404; bad
   CSRF → 400; total never negative (manual-line rule enforced).

## Non-goals

No payments (GL9 — `paid` transitions arrive there; this task only treats
`paid` as a frozen terminal state it must respect), no tax, no Stripe, no
PDF, no overdue logic, no real email. Do not touch anything outside
`pilot/greenlane/`.
